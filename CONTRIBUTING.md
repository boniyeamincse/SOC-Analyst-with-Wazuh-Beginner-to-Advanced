# Contributing

Thank you for contributing to this repository.

This project is open source and documentation contributions are welcome. You do not need to add code to make a valuable contribution. Clear explanations, better examples, diagrams, screenshots, link fixes, and lab corrections all improve the learning experience.

## Good Contribution Types

- Fix broken links or incorrect file references
- Improve grammar, clarity, and beginner readability
- Add diagrams, screenshots, and architecture explanations
- Update outdated commands or version-specific instructions
- Expand missing labs in Modules 2, 3, 4, and 6
- Add quizzes, checklists, and study notes

## Before You Start

1. Read the main [README.md](./README.md).
2. Check the target module README before editing module content.
3. Prefer small, focused pull requests.
4. Keep changes consistent with the current structure and naming.

## Contribution Workflow

### 1. Fork The Repository

Create your own fork of this repository on GitHub.

### 2. Create A Branch

Use a descriptive branch name.

Examples:
- `docs/module-2-agent-guide`
- `fix/broken-readme-links`
- `content/module-4-hunting-lab`

### 3. Make Your Changes

Keep the scope clear. One pull request should usually focus on one of these:
- one module
- one documentation problem
- one lab improvement
- one link or navigation cleanup

### 4. Review Your Work

Before opening a pull request, check:
- links are correct
- headings are clear
- markdown renders properly
- commands are formatted in fenced code blocks
- file names and paths match the real repository structure

### 5. Open A Pull Request

Your pull request should explain:
- what changed
- why it changed
- which module or files were updated
- whether you tested links or markdown rendering

## Documentation Standards

Follow the full documentation style guide in [docs/DOCUMENTATION-GUIDE.md](./docs/DOCUMENTATION-GUIDE.md).

Short version:
- write for beginners first
- explain why a topic matters, not only how
- use simple examples before advanced ones
- avoid large walls of text
- prefer accurate, current steps over broad but outdated coverage

## Recommended Pull Request Format

Use this template in your PR description.

```md
## Summary
- 

## What Changed
- 

## Validation
- Checked markdown formatting
- Verified local links

## Notes
- 
```

## Content Quality Checklist

- [ ] The explanation is easy for a beginner to follow
- [ ] The commands are readable and formatted correctly
- [ ] The links point to real files or valid external sources
- [ ] The content matches the current repo structure
- [ ] The examples are realistic and relevant to SOC learning

## Areas That Need Help

The repository especially needs help with:
- Module 2 labs
- Module 3 expanded rule-writing content
- Module 4 labs and supporting resources
- Module 6 practical scaling content
- more screenshots and workflow diagrams across the repo

## Respect Existing Work

Please avoid unrelated large rewrites in the same pull request. If you want to redesign a whole module, open a focused PR for that module only.
